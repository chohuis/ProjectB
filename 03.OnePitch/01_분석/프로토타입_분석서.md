## 0. 요약 (Executive Summary)

- **무엇인가**: 고교~프로 은퇴까지 투수 1명의 야구 인생을 육성·기록하는 1인 시뮬레이션. FM(커리어/기록/선택) + 롤러코스터 타이쿤2 샌드박스 감각. **엔딩은 판정이 아니라 기록(히스토리)이다**가 코어 철학.
- **어떻게 만들어졌나**: 3계층 — Svelte(UI, 렌더링만) / Electron(껍데기, IPC·파일I/O) / **Rust DLL(게임 본체 — 로직·난수·암호화 전부)**. 코드 규모 Rust ~10.9K줄, 핵심 TS ~5.2K줄.
- **핵심 설계 자산**: ① 주인공 반경 시뮬(멀수록 해상도↓), ② slot.db 단일 정본 데이터 아키텍처(파일=슬롯 SQLite + Repository 계층), ③ 육성 코어(능력치 9종·XP·구종·매치엔진·부상·S~D평가).
- **현 상태**: 초기 프로토(16,155명 사전생성·NPC상태 6곳 분산)의 구조 부채를 **git 342커밋 정량분석**으로 진단하고 "Lite 경량화" 리팩터링 R0~R5로 청산 중. R0~R4 완료, R5 진행 중(트레이드/콜업 캐던스·팜리그 시뮬 제거 완료, ABL/JBL Lazy·PendingAction 통합·히스토리 화면 미완).
- **정식버전 관점 결론**: 게임 설계(반경 시뮬·데이터 아키텍처·육성 코어)는 **성숙했고 계승 가치가 높다**. 반면 기술 스택은 미결이며(§16 추천), 코드베이스는 리팩터링 잔재가 남아 정식버전은 **설계는 계승, 코드는 선별 이식**이 합리적.

## 2. 게임 정체성

원형 `01_Game_Concept.txt` + `DESIGN.md §1`에서 확정된 불변 철학:

1. **선수는 경기 단위로 평가받는다** — 경기마다 S~D 등급.
2. **그러나 인생(커리어)은 점수로 판정하지 않는다** — 엔딩은 판정이 아니라 기록(히스토리).
3. **정답 루트 없는 샌드박스** — 목표는 유저 내면에 있고, 게임은 과정과 기록을 제공.

- **장르**: 야구 투수 인생 시뮬레이션 (1인 육성)
- **핵심 방향**: 투수 중심 / 장기 커리어 / 방대한 로그 누적 / 프로 진입이 목표지만 진입 여부로 인생을 판정하지 않음
- **Lite 선언(DESIGN §1.1)**: "선수 1명의 게임이다. 세계는 주인공의 배경이지 주인공이 아니다." → 육성 깊이는 유지, 주인공과 무관한 세계 연산은 축소.

> **정식버전 시사점**: 이 3대 철학은 정식버전에서도 불변 코어로 계승해야 한다. 특히 "기록=엔딩"은 원형부터 일관된 정체성인데 프로토에서 아직 화면으로 구현되지 않은(R5 미완) 최대 미완성 영역이다.

### 3.3 IPC — 단일 채널 `engine:call` (R2에서 확립)

초기 설계는 새 Rust 함수마다 **6단계 등록**(rs 작성 → lib.rs export → build → main.cjs 핸들러 → preload.cjs 브릿지 → projectb.d.ts 타입)이 필요했다. 이 등록 비용이 fix 커밋 핫스팟(preload+d.ts 82회)이었다.

R2에서 `engine:call(fn, payload)` 범용 핸들러로 통합 → **2단계**(Rust 작성 + `npm run build:native`)로 축소. 화이트리스트 = engine-native export 그 자체. 순수 포워딩 핸들러 81개 제거(92→28).

```typescript
// 신규 Rust 함수 호출 — 등록 없이 바로
const result = JSON.parse(
  await window.projectB!.engine("myFuncNative", JSON.stringify({ someValue: 42 }))
) as MyResultType;
```

> **정식버전 시사점**: 3계층 분리와 "본체를 네이티브에" 원칙은 상용 배포 시 변조 방지에 유효한 자산이다. 단 §16에서 논하듯 플랫폼 선택에 따라 Electron/napi-rs 조합의 유지 여부가 갈린다. `engine:call` 단일 채널 패턴은 어떤 스택으로 가도 계승할 설계 교훈.

### 4.4 데이터 3종류만 존재 (사전 생성 폐기 후)

| 종류 | 저장소 | git 관리 | 수정 방법 |
|---|---|---|---|
| ① 정의(콘텐츠) | master.db (read-only) | O — 소스 JSON만 | staging/ → deploy |
| ② 상태(세이브) | slot.db (슬롯별) | X | 런타임 repo 커맨드만 |
| ③ 파생(캐시) | 메모리 스토어 | X | 저장 안 함 |

**핵심 전환**: 선수 16,155명 사전 생성 JSON을 전부 폐기하고, **런타임에 Rust `generate_league_roster`가 worldSeed 결정적으로 생성**. "생성 규칙"만 git 관리 → 규칙 수정 = 즉시 패치. 사후 패치 스크립트(fix-ages·rebalance_ovr 등 15+개)가 원천 소멸.

```
새 게임 → Rust generate_league_roster(리그, 규칙, 이름풀, worldSeed) → slot.db npc INSERT (트랜잭션 1개)
매 시즌 → 신입생·드래프트 풀 필요 시점 Rust 생성 → INSERT
리그 활성화 → (ABL/JBL 진출 확정 시) 그 시점 로스터 생성
```

### 4.5 세이브 = 슬롯당 SQLite 파일 1개

- "단일 DB + 모든 테이블에 slot_id 컬럼" 폐기 → **파일 1개 = 세이브 1개**. 백업/공유/삭제 = 파일 복사/삭제. 슬롯 간 오염 원천 차단.
- 5그룹 구성: 메타 / 주인공(1행) / 선수(npc) / 시즌(휘발) / 기록(영구·append-only).
- **저장 버튼 개념 소멸**: repo 커맨드 = 트랜잭션 = 즉시 영속. 주 진행 1회 = 커밋 1회. "저장 안 하고 종료해서 유실" 클래스 소멸. 자동 롤링 백업(시즌 시작마다, 최근 3개).

### 4.6 성적 데이터의 생애 (해상도별 보존)

```
시즌 중  경기 시뮬 → season_stats 주간 누적 (반경1 리그만 개인 성적 존재)
시즌 종료 ① 주인공: careerRecord 확정 — 시즌 라인 + 경기별 gameLog 전체 영구 보존 (절대 압축 안 함 — 엔딩 코어)
         ② Named NPC: 연도 라인으로 압축, 경기 로그 폐기
         ③ 배경 NPC: 연도 라인만
         ④ 드리프트 리그: 팀 순위 + "리그 리더"(연감/뉴스용)만
         ⑤ season_* 리셋 → 다음 시즌
```

> **정식버전 시사점**: 이 데이터 아키텍처는 프로토의 **가장 값진 자산**이다. git 342커밋의 실패에서 역설계된 것이라 "왜 이렇게 해야 하는가"의 근거가 문서화돼 있다. 정식버전이 다른 스택으로 가더라도 (1) 단일 정본, (2) 능력치 동거, (3) Repository 커맨드형 변이, (4) 규칙 기반 런타임 생성, (5) 파일=슬롯 — 이 5원칙은 스택 무관하게 이식할 설계 교훈이다. ⚠ **미구현**: 세이브 무결성(HMAC) v3 대체 기능이 아직 없음 — 정식버전 착수 시 초기 설계 필요.

## 5. 핵심 설계 — 주인공 반경 시뮬레이션 (Radius Simulation)

> **주인공에게서 멀수록 해상도를 낮춘다.** — Lite 경량화의 중심 원리.

| 반경 | 대상 | 해상도 |

| **0** | 주인공 | 풀 — 스탯 9종·XP·구종·부상 18종·계약·평가 전부 |

| 독립 | 독립 8팀 | KBL | ABL·JBL·대학 |
| KBL | KBL 8팀 | ABL·JBL | — |
| ABL/JBL 진출 | 진출 리그 12팀 | KBL + 나머지 | — |

### 5.2 합성 주간 성적 생성기 (R3b 완료 — `synthetic_trajectory.rs`)

타 리그 Named NPC의 주간 성적을 라이벌 서사용으로 생성:
- **궤적 우선**: 시즌 시작 시 NPC별 최종 라인 목표 + 폼 곡선을 worldSeed로 사전 결정 → 주간 값은 궤적 보간. (주간 난수 누적이 아니라 궤적 보간이므로 시즌 합산이 항상 자연스러움)
- **결정적**: `(worldSeed, npcId, seasonYear, week)`만으로 매번 재계산하는 순수 함수 — 저장 불필요.
- **분포 안정**: 투수 `targetEra = clamp(6.5-(ovr-20)*4.5/70, 1.8, 7.5)`, 타자 `targetAvg = clamp(0.180+(ovr-20)*0.150/70, ...)`. 10시즌 표준편차 검증 통과.
- **합성 부상**: 궤적에 결장 구간으로 반영("출전 없는 부상" 방지).

> **정식버전 시사점**: 반경 시뮬은 "1인 육성 게임에서 세계를 어떻게 저렴하게 살아있게 할 것인가"라는 근본 문제의 성숙한 해답이다. 특히 합성 궤적 생성기(결정적·저장불필요·분포안정)는 영리한 설계. 정식버전의 규모/성능 목표를 정할 때 이 반경 정책이 핵심 축.

※ 실사용 17종 확인(DESIGN 표는 stale) — push/route/consume 로직이 MainPage·runAutoAdvance·개별 모달에 흩어져 R5 별도 세션 권장.

> **정식버전 시사점**: 커리어 흐름은 원형부터 확립된 게임의 골격. PendingAction(진행 정지 → 유저 처리 → 재개)은 이 게임의 상호작용 리듬 그 자체다. 정식버전에서 이 상태기계를 명시적으로 재설계하면(현재는 흩어져 있음) 큰 품질 향상.

## 14. 강점 / 약점 / 리스크 평가

### 강점 (계승 가치 높음)

1. **성숙한 데이터 아키텍처** — 342커밋 실패에서 역설계된 5원칙(§4). 근거가 문서화됨.

3. **깊은 육성 코어** — 능력치·구종·매치·부상·성장이 Rust로 이미 구현(§7).
4. **변조 방지 아키텍처** — 본체를 네이티브에 두는 3계층(§3).
5. **일관된 정체성** — 원형부터 "기록=엔딩" 철학이 흔들리지 않음.
6. **뛰어난 문서화** — DESIGN.md가 결정의 "왜"까지 기록. 인수인계 가능.

### 약점 (정식화 전 해결 필요)

1. **엔딩(히스토리 화면) 미구현** — 게임 정체성의 본체가 아직 화면이 없음.
2. **리팩터링 미완** — R5 잔여, 전환기 shim/adapter 잔존.
3. **세이브 무결성 공백** — HMAC v3 미구현.
4. **거대 파일** — game.ts/advanceWeek.ts 경계 추가 정리 필요.
5. **콘텐츠 볼륨 부족** — 캐릭터 2명·이벤트 소수 등 프로토 수준.
6. **아트/사운드/UX 미성숙** — 프로토 UI, 배경 gif 3장 수준.

### 리스크

- **스택 종속** — Electron+napi-rs는 데스크톱 배포엔 좋으나 웹/모바일 확장 시 재작성 큼.

| **2** | **육성 코어 알고리즘**(§7) | 게임의 전문성·차별점. Rust ~3.4K줄 이미 구현 | 스택 유지 시 그대로, 변경 시 로직 포팅 |
| **3** | **반경 시뮬 + 합성 궤적**(§5) | 1인 게임 세계 비용 해법 | 설계+Rust 코드 이식 |
| **4** | **커리어 프레임워크 + 히스토리 철학**(§8,§11) | 원형부터의 정체성 | 설계로 이식, 화면은 신규 |

### 15.2 재설계 후보

- **PendingAction 상태기계** — 현재 여러 파일 분산. 정식버전은 명시적 상태기계로 재설계.
- **게임 루프(advanceWeek)** — weekPhases 분해는 됐으나 정식버전 착수 시 경계 재정의 권장.
- **엔딩/히스토리 화면** — 완전 신규 설계(데이터는 존재).
- **UI/UX 전반** — 프로토 UI를 정식 품질로 재설계.

### 15.3 폐기 후보

- 전환기 shim/adapter(`npcAdapter.ts`, 레거시 채널 호환 래퍼) — 정식버전은 클린 스타트.
- 사전 생성 선수 JSON·관련 스크립트(이미 프로토에서도 폐기 진행).
- 레거시 ID 포맷 방어 변환(잔존 CLUB_/PKT_ 처리).
- master_overlay·JSON 블롭 잔재.

## 16. 기술 스택 추천 (미결정 → 분석 기반 권고)

사용자 결정 대기. 아래 3안 비교.

### A안 — 프로토 계승 (Svelte + Electron + Rust DLL) ★ 안정 우선 권고

- **장점**: 검증된 3계층, Rust 육성코어·매치엔진 재작성 없음, 변조 방지, slot.db 그대로. 착수 즉시 개발 가능.
- **단점**: 데스크톱 한정, Electron 번들 크기, 웹/모바일 확장 시 재작성, 단독 3스택 유지보수.
- **적합**: 데스크톱(Steam/자체배포) 정식 출시가 목표이고, 검증된 자산을 최대 계승하려는 경우.

### B안 — 웹/크로스플랫폼 재편 (엔진 로직 계승 + Electron 의존 축소)

- **방법**: Rust를 **WASM**으로 컴파일해 웹/데스크톱 공용 엔진화. UI는 Svelte 유지 또는 웹 표준. 세이브는 IndexedDB/OPFS 또는 서버.
- **장점**: 웹 배포 + 데스크톱(Tauri) 동시. Rust 자산 계승. Electron 탈피(번들↓).
- **단점**: WASM 포팅 비용(napi-rs → wasm-bindgen), 파일=슬롯 SQLite 재설계(OPFS SQLite/wa-sqlite), 변조 방지 약화(클라 WASM).
- **적합**: 웹 접근성·크로스플랫폼이 우선이고 데스크톱 변조방지를 일부 양보 가능한 경우.

### C안 — 모바일/앱 (03.App 방향)

- **관찰**: `03.App/`에 console.md·subscription.md·release-note.md 존재 → 앱스토어·구독형 힌트.
- **방법**: 코어 로직(Rust)만 계승, 프레젠테이션·수익화는 모바일 네이티브/크로스(Flutter·RN·Godot 등) 재설계.
- **장점**: 시장·수익화 잠재력. 구독 모델.
- **단점**: 사실상 재개발 규모. 프로토 UI 자산 대부분 폐기. 인터랙티브 매치 UI 터치 재설계.
- **적합**: 모바일이 최종 타깃이고 프로토를 "설계 레퍼런스"로만 쓸 경우.

### 권고 요약

| 목표 | 권고 |
|---|---|
| 빠른 정식 출시·검증 자산 최대 활용 | **A안** |
| 웹 접근성 + 데스크톱 동시 + Rust 계승 | **B안** (Tauri+WASM) |
| 모바일·구독 수익화 최종 타깃 | **C안** (코어만 계승) |

**단독 개발/빠른 정식화라면 A안**을 기본으로, 웹 확장 요구가 생기면 B안(Tauri 전환)이 자연스러운 진화 경로. 어느 안이든 **§4 데이터 5원칙과 §7 육성 코어는 계승**한다.
