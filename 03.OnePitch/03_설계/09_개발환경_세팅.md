# 개발 환경 세팅 가이드

> 근거: [01_환경](01_환경.md) §3-2·§6(스택·CLI) · [08_P7_체크리스트](08_P7_체크리스트.md)(이 절차로 실측 검증) · 대화 설계(2026-07-14, 실제 PC에서 세팅하며 확정)
> 상태: **Windows 기준 확정** — 이 절차대로 세팅한 환경에서 P7 전 항목(Windows PC 트랙) 실측 통과
> 목적: **다른 PC에서 개발을 시작할 때, 이 문서 순서대로 설치·확인하면 이 프로젝트가 검증한 환경과 동일한 상태를 재현**한다. 새 PC마다 이 시행착오를 반복하지 않기 위한 문서.

## 1. 필요 도구 · 검증된 버전

| 도구 | 검증된 버전 | 확인 명령 |
|---|---|---|
| Flutter | 3.38.4 (stable) | `flutter --version` |
| Rust / Cargo | 1.95.0 | `rustc --version` · `cargo --version` |
| JDK | Microsoft Build of OpenJDK 17.0.19.10 | `java -version` |
| Android SDK Platform | 34(필수) — Flutter가 36도 요구할 수 있음 | `flutter doctor -v` |
| Android Build-Tools | 34.0.0(필수) — Flutter가 28.0.3도 요구할 수 있음 | 〃 |
| Android NDK | 27.3.13750724 이상(Gradle이 빌드 시 28.x대를 선호해 자동 설치할 수 있음 — 정상) | 〃 |
| cargo-ndk | 4.1.2 | `cargo ndk --version`(단독 실행은 에러, `cargo ndk <subcmd>`로 사용) |
| flutter_rust_bridge_codegen | 2.12.0 | `flutter_rust_bridge_codegen --version` |
| flutter_steamworks (PC 배포용) | 0.0.4 | `flutter pub deps` |

- 버전은 **최소 기준선**이지 고정 상한이 아니다 — Flutter/Gradle이 더 최신 하위 컴포넌트를 요구하면 따라가면 된다(§3 참고).

## 2. 설치 순서 (Windows 기준, 2026-07-14 실제 검증 절차)

```
1. Flutter SDK 설치(공식 zip, PATH 등록) — flutter --version으로 확인
2. Rust 설치(rustup) 후 Android 타깃 4종 추가:
   rustup target add aarch64-linux-android armv7-linux-androideabi x86_64-linux-android i686-linux-android
3. JDK 17 설치:
   winget install --id Microsoft.OpenJDK.17 --accept-source-agreements --accept-package-agreements --silent
   setx JAVA_HOME "<설치경로, 예: C:\Program Files\Microsoft\jdk-17.x.x.x-hotspot>"
4. Android SDK cmdline-tools 설치:
   a. https://developer.android.com/studio#command-line-tools-only 에서 Windows용 zip 다운로드
   b. <sdk_root>\cmdline-tools\latest\ 구조로 압축 해제 (sdkmanager가 요구하는 필수 레이아웃 — latest 하위에 바로 bin/lib 등이 와야 함)
   c. 라이선스 전체 동의: sdkmanager --licenses
      (PowerShell 파이프로 y를 흘려보내면 씹힐 수 있음 — §3 참고. cmd /c "... < 답변파일.txt" 방식 권장)
   d. 필수 패키지 설치:
      sdkmanager "platform-tools" "platforms;android-34" "build-tools;34.0.0" "ndk;27.3.13750724"
   e. 환경변수: setx ANDROID_HOME "<sdk_root>" / setx ANDROID_SDK_ROOT "<sdk_root>"
   f. flutter config --android-sdk "<sdk_root>"
   g. flutter doctor -v 로 Android toolchain 확인 — 추가로 SDK 36 / Build-Tools 28.0.3을 요구하면:
      sdkmanager "platforms;android-36" "build-tools;28.0.3"
5. cargo install cargo-ndk
6. cargo install flutter_rust_bridge_codegen
7. flutter doctor -v 최종 확인 — Android/Windows 항목 "No issues found!" 목표
```

## 3. 이번 세팅에서 실제로 겪은 이슈 (재현 방지용)

| 이슈 | 증상 | 해결 |
|---|---|---|
| PowerShell 5.1 진행률 표시줄 버그 | `Invoke-WebRequest`로 대용량 파일 받을 때 비정상적으로 느림(150MB에 수 분 이상) | 다운로드 전 `$ProgressPreference = 'SilentlyContinue'` 설정 |
| `sdkmanager --licenses`가 PowerShell 파이프의 `y` 입력을 못 받음 | "review licenses" 프롬프트에서 멈춤 | PowerShell 파이프 대신 `cmd /c "sdkmanager --licenses < 답변파일.txt"`로 stdin 리다이렉션 |
| Windows 경로 길이 제한(MAX_PATH 260자) | 프로젝트가 깊은 경로(임시폴더 등)에 있으면 CMake/MSBuild가 중간 산출물 경로를 못 만들어 빌드 실패 | 프로젝트를 짧은 경로에 둘 것. 실제 리포지토리 구조(`app/`+`engine/`)는 짧은 경로라 무관 |
| 비영어 로케일(코드페이지 949) Windows에서 `flutter_steamworks` 0.0.4 빌드 실패 | MSVC가 패키지의 BOM 없는 UTF-8 C++ 소스를 시스템 코드페이지로 오인식 → 컴파일 에러 | 프로젝트 `windows/CMakeLists.txt`의 `APPLY_STANDARD_SETTINGS` 함수에 `/utf-8` 컴파일 옵션 추가(패키지 자체는 안 건드림) — **실구현 시 프로젝트 표준 CMake 설정에 포함할 것** |
| `flutter test`가 네이티브 `.dll`을 못 찾음 | 헤드리스 Dart 테스트 실행 시 `Failed to load dynamic library` | `flutter build windows` 후 산출된 `.dll`을 프로젝트 루트로 복사(테스트 실행 시에만 필요, 실제 앱 실행/배포는 빌드 시스템이 자동 배치하므로 무관) |
| Gradle 첫 Android 빌드가 느림 | NDK 28.x·Build-Tools 35·Platform 33·CMake를 프로젝트 요구에 맞춰 자동으로 추가 다운로드 | 정상 동작 — 사전에 정확한 버전을 다 맞출 필요 없음. 단 첫 빌드는 네트워크·시간이 더 걸림(약 15분 경험) |
| 프로젝트 절대경로에 한글(예: `업무폴더`·`01.기획`)이 있으면 `flutter build windows` 실패 | cargokit `run_build_tool.cmd`가 빌드 시점에 `cmd.exe echo`로 임시 `pubspec.yaml`을 생성하는데, 시스템 코드페이지(949)가 유니코드 경로를 mojibake시켜 `build_tool` 패키지 경로를 못 찾음(`Couldn't resolve the package 'build_tool'`) | **리포지토리 전체를 영문(ASCII) 경로로 이동**해 해결(2026-07-15). `cargo build`(엔진 단독)는 한글 경로와 무관하게 정상 — cargokit의 Windows 배치 스크립트에서만 발생. **경로 이동 후 `app/build/`(CMake 캐시)가 이전 경로를 참조해 `CMakeCache.txt directory ... is different` 에러가 남 — `app/build/` 삭제(gitignore 대상, 안전) 후 재빌드하면 해결** |

## 4. 새 PC 세팅 확인 체크리스트 (빠른 검증)

새 PC에서 위 §2를 마친 뒤, 아래를 한 번에 실행해 이 프로젝트가 검증한 환경과 동일한지 확인한다.

```
flutter --version
rustc --version && cargo --version
java -version
flutter doctor -v
cargo ndk --version
flutter_rust_bridge_codegen --version
```

- `flutter doctor -v`에서 Android·Windows(또는 해당 OS) 항목에 `X`(치명 오류)가 없으면 통과.
- 전부 통과하면 [08_P7_체크리스트](08_P7_체크리스트.md)를 처음부터 다시 돌릴 필요 없이 **바로 구현 착수**(엔진 스캐폴딩)로 진입 가능.
- 하나라도 실패하면 §3 표에서 동일 증상이 있는지 먼저 확인.

## 5. 참고 · 한계

- 이 절차는 **Windows 11에서 실제로 실행해 검증**됨(2026-07-14, [08_P7_체크리스트](08_P7_체크리스트.md)).
- **Mac/Linux는 미검증** — 해당 OS에서 개발을 시작하는 시점에 이 문서에 OS별 절차를 추가해야 한다(§6).
- Android **실기기/에뮬레이터 동작**, **Steam 클라이언트 실연동**은 이 절차(툴체인 설치)로는 확인되지 않는다 — 별도로 실기기·Steam 클라이언트 설치가 필요([08_P7_체크리스트](08_P7_체크리스트.md) §6 열린 세부).

## 6. 열린 세부

- [ ] Mac/Linux 개발환경 세팅 절차 — 해당 OS에서 개발 착수 시 추가
- [ ] Android Studio(GUI) 설치 경로의 대안 절차 문서화 여부 — 현재는 cmdline-tools만 검증·문서화
