## 1. 스택 결정

- **타깃**: PC(Windows/Mac/Linux) + **Android**. web 제외.
- **UI**: **Flutter (Dart)** — 가장 검증된 PC+Android 단일 코드베이스.
- **엔진**: **Rust 네이티브 라이브러리** — 게임 본체(로직·난수·시뮬·데이터·암호화).
- **경계**: **flutter_rust_bridge (FFI)** — Dart↔Rust 타입 자동 생성.
- 기존 Svelte UI·Rust 로직은 **참고 모티브로만**, 재작성.

## 2. 계층 구조

| 레이어 | 역할 | 금지 |
|---|---|---|
| **Flutter (Dart)** | 화면 렌더·입력·화면 상태·애니메이션(투구=CustomPainter/Flame) | 게임 로직, 난수 |
| **flutter_rust_bridge** | Dart↔Rust FFI 경계 (코드젠 타입드 바인딩) | — |
| **Rust 엔진 (네이티브 lib)** | 로직·난수·시뮬 + slot.db(rusqlite) + content.db 로드 + 암호화/HMAC | — |

> 계층 간 계약(누가 무엇을 하는가)은 [03_구조](03_구조.md) 참조.

## 3. 왜 이 조합인가

- **성숙도·자료 축적 최상** — Flutter 커뮤니티·패키지·SQLite 도구 방대.
- **WebView 파편화 없음** — Flutter 자체 렌더러(Skia/Impeller) → System WebView 버전 의존 소멸(Tauri/Capacitor 대비 안정성 이득).
- **Rust 네이티브 유지** — 성능 + 변조방지 옵션 보존(D4-2 은닉 밸런스). 프로토 Rust를 재구성 모티브로.
- **단일 코드베이스** — Rust는 Android `.so`(cargo-ndk) / 데스크톱 lib. Flutter가 양쪽 껍데기.

### 이 선택이 해소한 항목

| 항목 | 해소 |
|---|---|
| 인터랙티브 매치 상태 | 매치 세션을 Rust에 보유(frb 불투명 핸들) |
| 무거운 시뮬 UI 프리즈 | frb async + Rust 별도 스레드 → 논블로킹 |
| WebView 버전 파편화 | 소멸(Flutter 자체 렌더) |
| 프로토 IPC 6단계 등록비용 | frb 코드젠 자동 생성 |

### 3-1. 재검토(2026-07-14): Flame 여부 · PC 게임으로서 Flutter 적합성

사용자 요청으로 스택을 한 번 더 검증. 결론 = **유지**.

- **Flame 미채택 이유는 Rust DLL 호환성이 아니다.** Flame은 별도 엔진이 아니라 Flutter 위 Dart 패키지(게임루프+캔버스)라 frb의 Rust 호출과 레이어가 겹치지 않음 — 애초에 충돌 지점 없음. 미채택 이유는 **장르**: 매치 화면이 이닝별 텍스트 요약+경량 상황판+가끔 1구 3x3 코스그리드뿐이라 스프라이트·충돌·카메라 등 Flame 고유 기능을 쓸 일이 없음(→ [04_UI기획/05_매치](../04_UI기획/05_매치.md) 결정 6, CustomPainter로 충분). 특정 화면에 국소적으로 Flame을 얹는 것 자체는 이후에도 열려있음(전면 채택/미채택 문제 아님).
- **Flutter가 "PC 게임"에 맞는가**: 판단 기준=장르. 이 게임은 텍스트·리스트·폼·다이얼로그 중심 매니지먼트 시뮬(액션·물리·파티클 없음) — 이런 장르는 게임엔진보다 앱 프레임워크가 유리(리스트 가상화·반응형 레이아웃·폼 처리). Flutter Desktop은 3.x부터 stable, 프로덕션 데스크톱 앱 사례 다수(단, "진짜 PC 게임" 사례는 적어 리스크로 인지). 액션성 강한 장르였다면 Unity/Godot이 맞았을 것.
- **Steam 배포 확정(사용자 결정, 2026-07-14)** — 최초엔 `steamworks-rs`로 Rust 쪽 우회 연동을 전제했으나 §3-2 재조사로 **`flutter_steamworks`(Dart 패키지, Win/Mac/Linux 지원, Flame 불필요)로 정정** — DIY FFI 배관 없이 기성 패키지 사용. **P7 스파이크 범위에 포함**(§7).

### 3-2. 경쟁 후보 전수 비교(2026-07-14) — 최종 재확정

사용자 요청으로 Electron·Tauri2·Godot·React Native·Web(WASM)·네이티브분리까지 전부 놓고 "PC단독/Android단독/PC+Android동시" 3가지 시나리오로 재비교(웹검색으로 2026년 최신 상태 확인).

| 후보 | PC 단독 | Android 단독 | PC+Android 동시(우리 요구) |
|---|---|---|---|
| Electron+Rust(프로토 원안) | 양호 | **불가**(모바일 미지원) | 탈락 |
| Tauri2+Rust | 최상급(가장 가벼움, 2026 프로덕션급) | 2026년 프로덕션 준비 단계로 성숙했으나 Android WebView 파편화는 OS단 구조적 리스크 잔존 | 준수하나 WebView 편차 리스크 |
| **Flutter+Rust(frb)** | 양호(`flutter_steamworks`로 Steam 리스크 낮음) | **최상급**(frb+Android가 가장 검증된 조합, WebView 편차 없음) | **최상급 — 채택** |
| Godot+Rust(gdext) | 최상급(GodotSteam 매우 성숙) | 양호하나 **gdext 자체의 Android 지원이 "experimental, tooling 부족"** — 우리처럼 로직 전부를 Rust에 두는 구조엔 리스크 | 강하나 정확히 우리에게 필요한 지점(Rust+Android)이 실험단계 |
| React Native+Rust | 약함(Linux 미지원) | 양호 | 탈락(Linux 없음) |
| Web/WASM | 가능하나 slot.db 파일기반 SQLite 설계 전체 재설계 필요(WASM은 파일시스템 직접접근 불가) | 동일 | 탈락급 |
| 네이티브 분리(WinUI/GTK+Compose) | 최상 | 최상 | 기술적 최상이나 UI 코드베이스 2벌 영구 유지 — 소규모 팀엔 과함 |

**결론**: Flutter+Rust(frb) **최종 확정**. Steam 연동은 `flutter_steamworks`로 낮은 리스크임이 확인돼 Godot 대비 유일한 약점이 해소됐고, 반대로 Godot의 약점(gdext Android 실험단계)은 우리 요구와 정면충돌. Electron·RN·Web은 이번 비교로 명확히 배제.

## 4. 플랫폼 세부

- **저장·백업**: 위치 = PC OS 앱데이터 / Android 앱 내부저장소(권한 불필요), `path_provider` 일원화. **로컬 자동 롤링 백업만**(시즌 시작 3개 + 마이그레이션 전 백업). 수동 내보내기·클라우드 동기화 미채택(추후 가능).
- **콘텐츠 번들**: content.db 앱 번들(read-only). Android는 첫 실행·`content_version` 변경 시 내부저장소로 **추출 후 오픈**, PC는 리소스 직접 오픈 → 경로 일원화.
- **업데이트**: 앱 업데이트 = PC 스토어/인스톨러 · Android Play. 콘텐츠는 앱 업데이트에 실림. `content_version`으로 콘텐츠 OTA 미래 가능(서버 미구축). **PC 배포채널 = Steam 확정(2026-07-14, §3-1), 그 외(모바일 채널 세부·수익화)는 기획 비즈니스 결정으로 이월**.

## 5. 배포

- PC: Flutter desktop 번들(Windows/Mac/Linux) + **Steam**(`flutter_steamworks` 패키지로 Dart 레이어에서 직접 연동 — P7에서 실측, §7).
- Android: `.aab`/`.apk`(Play), Rust는 cargo-ndk로 `.so`.

## 6. 빌드 · 개발 툴 (단일 Dart CLI)

프로토는 npm script 40여 개 난립(deploy·gen:manifest·build:masterdb·build:native·test:* 7종…). 새 스택은 Electron·Vite·Node·콘텐츠 파이프라인 소멸로 이미 단순 → 남은 명령을 **단일 Dart CLI**(`dart run tool <cmd>`)로 통합.

| 서브커맨드 | 역할 |
|---|---|
| `gen` | frb 코드젠(Dart↔Rust 바인딩 자동생성) |
| `run pc\|android` | 개발 실행(gen + cargo + flutter run) |
| `build pc\|android` | 릴리스 빌드(cargo/cargo-ndk + flutter build) |
| `content seed` | `data/seed/*.csv·toml`(172팀·리그·구장·특성·생성규칙 등) → content.db, 전량 재구성(`--dry-run`) — [07_데이터관리](07_데이터관리.md) §3-2 |
| `content import <file>` | AI 생성 JSON → content.db (가산, `--dry-run`) |
| `content validate` | 스키마 + 참조무결성(FK) + 번역 커버리지 검사(seed 산출물 포함) |
| `content dump` | content.db → 텍스트 스냅샷(git 이력) |
| `test` | Rust + Dart 테스트 |
| `release` | 패키징(스토어/인스톨러) |

- 단일 언어(Dart) 툴링 — 앱도 툴도 Dart, 콘텐츠 도구까지 한 곳.
- 프로토 대비 소멸: `deploy`·`gen:manifest`·`build:masterdb`·`gen:contacts`·`gen:index`·`build:packages`.
