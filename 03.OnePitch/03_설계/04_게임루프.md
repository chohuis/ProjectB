## 1. 시간 계층 — 일/주/월/시즌 (정합)

기획은 "하루씩 진행"([01_커리어_구조](../02_기획/01_커리어_구조.md) §1)이지만 육성 처리는 대부분 주 경계 배치다 — 모순이 아니라 **진행 단위(일)와 처리 경계(주/월/시즌)가 다른 층위**임을 명시한다.

- **최소 진행 단위 = 1일.** 1시즌 = 52주 = 364일(표기 편의).
- 엔진은 하루씩 나아가되, 각 경계를 지날 때 해당 배치를 실행한다.

| 경계 | 처리 항목 | 근거 |
|---|---|---|
| **매일** | 스케줄된 경기 시뮬 · 콜백형 이벤트(사건 발생 즉시) · 급성형 부상(경기 중) | [07_매치_엔진](../02_기획/육성코어/07_매치_엔진.md) · [02_이벤트](../02_기획/콘텐츠/02_이벤트.md) §1 · [08_부상_시스템](../02_기획/육성코어/08_부상_시스템.md) §3 |
| **주 경계** | 훈련 XP→스탯 변환 · 누적형 부상 체크 · 라이브상태(피로·컨디션·사기) 갱신 · 폴링형 이벤트 평가 · 개인 트레이너 과금 | [06_훈련_시스템](../02_기획/육성코어/06_훈련_시스템.md) · [04_성장_곡선](../02_기획/육성코어/04_성장_곡선.md) §2 · [08_부상_시스템](../02_기획/육성코어/08_부상_시스템.md) §3 · [02_이벤트](../02_기획/콘텐츠/02_이벤트.md) §1 · [08_개인_재정](../02_기획/08_개인_재정.md) §3-2 |
| **월 경계** | 월 단위 평가("이달의 페이스") | [09_평가_시스템](../02_기획/육성코어/09_평가_시스템.md) §5-1 |

## 2. 엔진 진행 커맨드 — `advance()`

프로토의 `advanceWeek`를 대체하는 단일 진행 커맨드. [03_구조](03_구조.md) §5 커맨드 표면의 핵심.

```
advance() → PendingAction[]
```

- 엔진이 내부적으로 **다음 정지 지점까지 하루씩 진행**하며 §1의 경계 배치를 순서대로 실행 → 정지 시점의 PendingAction 목록 반환. UI는 "다음 정지까지 continue"만 호출(빈 날을 클릭으로 넘기지 않음 — [01_커리어_구조](../02_기획/01_커리어_구조.md) §1 "개입 지점에서 자동 정지").
- **정지 지점** = ① 새 PendingAction 발생(§3) ② 주인공 등판 경기 도달.
- **한 날에 여러 경계가 겹칠 때 처리 순서**: `경기 → 주간배치 → 월간배치 → 시즌배치`. 경기를 먼저 처리해야 그날 성적이 이후 배치(XP·평가·자책 등)의 입력이 된다.
- **결정성**: 모든 배치·시뮬은 worldSeed 기반([03_구조](03_구조.md) §3-5) — 같은 세이브에서 advance는 항상 같은 결과.

## 3. PendingAction — 진행 정지 타입 (명시적 재설계)

프로토는 16종 난립 → 8종 통합 시도 중이었으나 로직이 여러 파일에 분산돼 미완([분석서](../01_분석/프로토타입_분석서.md) §8.3). 정식은 **7종으로 통합·명시**한다.

**통합 원칙**: **전용 화면**([03_메시지_알림](../02_기획/콘텐츠/03_메시지_알림.md) §2-1)이 필요한 상호작용만 독립 타입, 나머지 "선택지 있는 사건"은 전부 `event`에 흡수(sub-type + 긴급도 플래그로 구분).

| 타입 | 무엇 | 전용 화면 | 흡수하는 기존 항목 |
|---|---|---|---|
| `game` | 주인공 등판 경기 | 매치 세션(클러치 딥다이브) | — |
| `event` | 선택지 있는 이벤트 전반 | 메시지함 인박스 내 처리 | 국대 발탁 · 시즌말 투자 · 사치품 · 부상경고 후속 · 방출경고 후속 · 스폰서 제안 · 병역 강제입영 |
| `careerChoice` | 갈림길 A/B 진로·병역경로 | 진로선택 화면 | 병역 경로 선택 |
| `draft` | 드래프트 지명 결과 통보 | 드래프트 결과 화면 | — |
| `contractNego` | 재계약·FA 협상(다단계) | FA·재계약 협상 화면 | — |
| `tradeDecision` | 트레이드 통보·거부권 행사 | 트레이드 오퍼 비교 화면 | — |
| `injuryTreatment` | 부상 치료법 선택 | 치료 선택 화면 | — |

- **주간 훈련 슬롯 배분**([03_메시지_알림](../02_기획/콘텐츠/03_메시지_알림.md) §2-1)은 PendingAction이 아님 — "지속 설정"([01_커리어_구조](../02_기획/01_커리어_구조.md) §1)이라 상시 접근하는 화면이지 진행을 막는 정지가 아니다.
- **기록보관소·은퇴 화면**도 PendingAction 아님 — 은퇴는 트리거([05_히스토리_엔딩](../02_기획/05_히스토리_엔딩.md) §3)로 진입하는 종료 흐름.

## 4. 상태기계 전이 (4단계)

각 PendingAction의 생애 — 로직을 한 곳(엔진)에 집중해 프로토의 "여러 파일 분산"([분석서](../01_분석/프로토타입_분석서.md) §8.3) 부채를 차단.

```
push (엔진 감지·생성) → present (UI 표시) → resolve (유저가 커맨드로 회신) → consume (엔진 상태변이 + 다음 advance)
```

- **긴급(블로킹)**: resolve 전까지 `advance()` 차단 — 콜백형(방출·계약 등) 대부분([02_이벤트](../02_기획/콘텐츠/02_이벤트.md) §6).
- **비긴급(논블로킹)**: 인박스에 대기, `advance()` 계속 진행 — 확률형 플레이버 등.
- 회신 커맨드: `resolveChoice(actionId, choiceId)`([03_구조](03_구조.md) §5 이벤트 커맨드) 등 타입별 커맨드로 consume.

## 5. 저장 연동

- [02_데이터](02_데이터.md) `slot.db` 시즌(휘발) 그룹의 `pending_actions` 테이블 — 각 행: `type`, `payload`(JSON), `urgency`, `created_day`. 세이브/로드 시 미해결 PendingAction이 그대로 복원돼 이어서 처리.
- **경기 중 상태는 pending_actions에 없음** — [07_매치_엔진](../02_기획/육성코어/07_매치_엔진.md) §12 "경기 중 저장 불가"와 일치. `game` PendingAction은 "경기가 예정됨"까지만 저장하고, 매치 세션 자체는 커밋되지 않는다.
