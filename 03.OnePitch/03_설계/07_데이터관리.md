## 1. 데이터 3덩어리 (성격·저장소·볼륨)

성격이 완전히 다르므로 관리 방식도 다르다.

| 덩어리 | 예 | 저장소(원본) | 원본 볼륨 | 저작 방법 |
|---|---|---|---|---|
| **① 고정 세계 참조** | 리그·172팀·구장·컬러·특성·히스토리 | content.db (§3) | **작음**(172행대) | 손저작(기획 확정분) |
| **② 생성 규칙·풀** | 스탯범위·포지션비율·이름풀·성격가중·**canonical_seed** | content.db (§3) | **작음** | 손저작(값은 D그룹) |
| **③ 대량 콘텐츠** | 이벤트·업적·캐릭터·서술템플릿·번역 | content.db | 큼 | 직접저작+AI import([02_데이터](02_데이터.md) §3) |

- **선수(NPC)는 어디에도 "원본으로 저장하지 않는다"** — §2 방식으로 생성. slot.db(세이브별)에만 존재.
- 주인공은 [캐릭터 생성 흐름](../04_UI기획/06_캐릭터생성.md)으로 만들어져 slot.db에.

## 2. 선수 데이터 = 고정 시드 생성

**결정(2026-07-14)**: 매 게임 랜덤이 아니라 **단일 canonical seed로 결정적 생성** → 모든 게임 동일 초기 로스터. 원본은 개별 선수가 아니라 **규칙 + seed 1개**만 저장(엑셀 "수식 저장" 방식 — 규칙을 바꾸면 로스터가 자동 추종).

### 2-1. 초기 로스터 생성 (새 게임 시)

```
새 게임 → Rust generateInitialWorld(generation_rules, name_pools, canonical_seed)
        → 172팀 로스터(~3,700명) + 스태프 결정적 생성 → slot.db npc/protagonist 이웃 테이블
```

- **모든 게임 동일** — 같은 규칙·seed → 같은 선수. "이 팀 에이스는 항상 이 선수"([리그팀/06_리그_히스토리](../02_기획/리그팀/06_리그_히스토리.md) 고정 세계관과 정합).
- **개별 손수정 없음** — 전원 절차적([콘텐츠/01_캐릭터](../02_기획/콘텐츠/01_캐릭터.md))과 일치. 특정 선수를 바꾸려면 규칙을 바꿈(규칙 단위).
- **밸런스와 자동 정합** — D그룹 규칙값을 조정하면 로스터가 자동 반영. 별도 동기화·재동결 불필요(← 프로토 대조 지옥 차단).

### 2-2. 이후 유입 (매 시즌, 여전히 생성)

커리어 15~20시즌 동안 신입생·드래프트로 계속 새 선수가 유입되므로 생성은 사라지지 않는다.

- **초기 로스터** = `canonical_seed`(고정, 전 게임 동일).
- **이후 유입** = 세이브별 `meta.world_seed`(새 게임 시 발급) 기반 파생 → **시작은 같되 진행하며 세이브마다 갈라짐**(트레이드·은퇴·주인공 선택으로 세계가 자연히 분기). "고정"은 **초기 상태**를 뜻하지 매 경기 결과까지 동일하다는 뜻이 아님(그건 재플레이성 0).
- 세이브 내에서는 결정적(world_seed) — 세이브/로드 재현 보장.

## 3. 고정 세계 참조 데이터 (172팀 등) — 초기 구성 · 원본 위치

현재 172팀 데이터는 기획 markdown([리그팀 문서](../02_기획/리그팀/00_개요.md))에 확정돼 있다(이름·연고·전력★·특성3슬롯·히스토리·컬러·구장). 이걸 content.db로 옮겨야 한다.

**추천(확인 필요)**: **git의 사람이 읽는 시드 파일이 원본 → 작은 seed 스텝이 content.db를 채움.**

- **이유**: ①·② 데이터는 **소량·고정·리뷰 가치 높음**(172팀 = 카논 로스터, PR에서 diff·리뷰하고 싶은 대상). 이미 markdown 표로 써뒀으니 그걸 구조화(CSV/TOML/JSON)해 시드 원본으로 승격하면 **markdown↔DB 이중관리(drift) 소멸**.
- **[02_데이터](02_데이터.md) "빌드 스텝 없음" 원칙과의 관계**: 그 원칙은 **대량 AI 콘텐츠(③)**의 파이프라인 고통 때문이었다. **소량 고정 refs(①·②)의 작은 결정적 시드**는 그 고통이 없다 — 둘을 구분 적용한다(③은 여전히 content.db 직접).
- **대안(content.db 직접)**: DB가 유일 원본, git엔 `dump` 스냅샷. 빌드 스텝은 없으나 172팀 표를 diff·리뷰하기 불편하고 markdown이 stale.

## 4. 대량 콘텐츠(③) — 02_데이터 그대로

- content.db **직접저작 + AI import(가산·dry-run) + dump(git 스냅샷)** — [02_데이터](02_데이터.md) §3 재확인. 변경 없음.

## 5. 수정 워크플로 (이후 데이터 변경)

| 변경 대상 | 절차 | 기존 세이브 영향 |
|---|---|---|
| **팀·리그·구장 등(①)** | 시드 파일 편집 → seed 재실행 → `content_version` bump | 신규 게임만 반영 |
| **생성 규칙(②, 밸런스)** | rules 편집 → 로스터 **자동 추종**(§2) | 신규 게임만 반영 |
| **콘텐츠(③)** | AI import(가산) 또는 DB 직접 편집 → `content_version` bump | [02_데이터](02_데이터.md) §5 마이그레이션 인지 |

- **핵심 원칙**: **이미 생성돼 slot.db에 있는 선수는 그 세이브의 정본** — content.db 규칙을 나중에 바꿔도 **진행 중 세이브의 선수를 덮어쓰지 않는다**(커리어 중간에 선수 스탯이 패치로 바뀌면 안 됨). 규칙 변경은 **다음 새 게임의 생성**에만 적용.
- `content_version`으로 세이브가 어떤 콘텐츠 버전에서 시작됐는지 추적([02_데이터](02_데이터.md) §5).

## 6. 볼륨 관리 (선수 대량 문제)

- **원본(git)**: 소량 — 172팀 refs + 규칙 + seed + 콘텐츠 소스. **선수 수천 명은 원본에 없음**(생성이므로). 프로토의 16,155 선수 JSON 부채 소멸([분석서](../01_분석/프로토타입_분석서.md) §4.4).
- **세이브(slot.db)**: 선수 수천 행 + 가변 — 이미 설계된 3중 장치로 관리:
  - **Repository 커맨드**(트랜잭션 1개, [02_데이터](02_데이터.md) §4).
  - **시즌 종료 압축**([분석서](../01_분석/프로토타입_분석서.md) §4.6): 주인공=전체 보존, Named NPC=연도라인, 배경 NPC=최소.
  - **반경 시뮬**([분석서](../01_분석/프로토타입_분석서.md) §5): 먼 리그는 순위/리더만.

## 7. 초기 구축 순서 (구현 시)

```
1. content.db 스키마 생성           ← 06_스키마 v1 DDL
2. 고정 세계 참조 시드 입력          ← 172팀·리그·구장·컬러·특성·히스토리 (§3)
3. 생성 규칙·이름풀·canonical_seed 입력
4. (D그룹 후) 규칙 값 채움           ← 05_밸런스 시뮬 하네스로 조정
5. 콘텐츠 AI import                 ← 이벤트·업적 등 (§4)
6. 새 게임 → generateInitialWorld → slot.db 검증
```
