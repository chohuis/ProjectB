# 데이터 (저장소 · 스키마 · 다국어)

> 결정 기록. 근거: [분석서](../01_분석/프로토타입_분석서.md) §4, §13 · [03_구조](03_구조.md) · 요약: [00_결정_요약](00_결정_요약.md)
> 상태: **확정(초안)** · 일자: 2026-07-09
> 변경 이력: 콘텐츠 저장 v1(JSON5 파일) → v2(content.db 직접). 저장 기술 일관성·참조무결성·"진짜 DB" 선호로 콘텐츠도 SQLite로.

## 1. 문제 인식 (프로토 콘텐츠 파이프라인)

프로토의 고통은 **SQLite 자체가 아니라** `JSON 수백개 → deploy → build:masterdb → master.db + gen:manifest` **파이프라인**이었다. JSON이 원본이고 DB가 산출물이라 대조가 필요했고, 흩어짐·느린빌드·꼬임·DB디버깅이 여기서 나왔다.

## 2. 원칙: 데이터는 세 곳, 전부 엔진 뒤

| 성격 | 저장소 | 가변성 | 저작 방법 |
|---|---|---|---|
| **정의(콘텐츠)** — 이벤트·대사·업적·이름풀·생성규칙·refs | `content.db` (SQLite, 번들, read-only) | 불변(런타임) | 직접 편집 + AI import |
| **상태(세이브)** — 선수·계약·성적·주인공 | `slot_<id>.db` (SQLite, 파일=슬롯) | 가변 | Repository 커맨드 |
| **은닉 밸런스** (D4-2) — 매치확률·성장계수·평가경계 | **Rust 상수**(컴파일 내장) | 불변 | 코드 |

세 곳 **전부 Rust 엔진 뒤**. UI는 직접 읽지 않는다([03_구조](03_구조.md)). **빌드·deploy·manifest 없음.**

## 3. 콘텐츠 = content.db (직접 저작)

- **원본이 곧 DB**: content.db가 source of truth. "JSON→컴파일 산출물"이 아님 → 대조 지옥 소멸.
- **중첩 서사 처리**: 조회·인덱스 필드(id·stage·week·type·참조ID)는 **평칼럼**, 분기 대사 등 복잡 구조는 **행 안 JSON(TEXT) 컬럼**.
- **참조 무결성**: FK로 없는 캐릭터/팀 참조를 DB가 자동 차단.
- **런타임**: 시작 시 Rust가 통째로 **메모리 구조체 로드** → 이후 HashMap 조회. DB는 저작·무결성·배포 계층.
- **번들**: Flutter 앱 에셋(Android asset / 데스크톱 리소스).

### AI 대량생성 워크플로

```
AI가 JSON 생성 (스키마/프롬프트 기준)
  → import --dry-run : 스키마 + 참조무결성 + 번역 커버리지 검증
  → import           : content.db에 행 가산(추가)
  → (선택) dump      : content.db → 텍스트 스냅샷(git 이력)
```

- 프로토와 다른 점: ① content.db가 원본 ② import는 **AI 대량추가 때만·가산적**, 손편집은 DB에 직접 ③ manifest·흩어짐 없음. 생성용 JSON은 버리는 입력.

### content.db 스키마 (초안)

**규칙**: 조회·FK 필드는 **평칼럼**, 중첩·가변은 **JSON(TEXT) 컬럼**.

| 테이블 | 평칼럼(조회/FK) | JSON 컬럼 |
|---|---|---|
| `events` | id, stage, week, type | trigger, title/lines/choices |
| `messages`/`decisions` | id, category | template |
| `achievements` | id, category | condition, meta |
| `characters` | id, name, role | arcs(대화·친밀도) |
| `name_pools` | id, locale, kind | names[] |
| `generation_rules` | league_id(FK) | rules(스탯범위·포지션비율) |
| `leagues`/`teams`/`schools`(refs) | id, (team)league_id(FK) | meta |
| `balance_open` | key | value (개방 밸런스만, D4-2) |
| `strings` | string_id, locale | text (표시 텍스트 번역 — §7) |

- **표시 텍스트는 `string_id` 키로 저장**, 번역은 `strings` 테이블(§7 다국어).
- FK로 참조 무결성 강제. import 시 전수 검증.

## 4. 세이브 = slot.db (프로토 v3 계승, 접근만 Node→Rust)

better-sqlite3(Node) → **rusqlite(Rust 엔진 내)**. 파일=슬롯(`slot_<id>.db`), 백업/삭제 = 파일 복사/삭제.

### slot.db 스키마 (5그룹)

| 그룹 | 테이블 | 핵심 |
|---|---|---|
| 메타 | `meta` | save_version, content_version, worldSeed, stage, playtime, integrity_sig |
| 주인공 | `protagonist`(1행) | 정체성 + stats(JSON)·pitches·contract·injury |
| 선수 | `npc` | id, name, team_id, position, age, is_named + **stats(JSON)·xp(JSON)**·form·personality·retired |
| 시즌(휘발) | `season_meta`·`schedule`·`standings`·`season_stats`·`pending_actions` | 시즌 종료 시 압축→리셋 |
| 기록(영구) | `career_history`·`game_log`·`league_transactions`·`history_standings`·`history_leaders` | append-only |

- **능력치 동거**: stats·xp는 JSON 1컬럼(스탯 추가 시 스키마 패치 불필요). 은퇴해도 npc 행 유지.
- 계승 5원칙: ①단일 정본 ②능력치 동거 ③Repository 계층 ④커맨드형 변이(트랜잭션 1개) ⑤규칙 기반 런타임 생성(worldSeed 결정적).

### Repository 커맨드 인벤토리 (초안)

각 커맨드 = SQLite 트랜잭션 1개, 실패 시 전체 롤백. 거래는 npc행 + `league_transactions` 원자 기록.

| 분류 | 커맨드 |
|---|---|
| 슬롯 수명주기 | `createSlot`·`loadSlot`·`migrateSlot`·`rollingBackup` |
| 로스터 생성(결정적) | `generateLeagueRoster`·`generateFreshmen` |
| 선수 변이 | `transfer`·`swapTeams`·`assignDraft`·`signFA`·`enlist`·`discharge`·`retire`·`updateWeekly` |
| 주인공 | `signContract`·`negotiateSalary`·`applyInjury`/`treat` |
| 진행 | `advanceWeek`(오케스트레이션)·`seasonRollover` |
| 이벤트 | `resolveChoice` |

## 5. 세이브 버전 · 마이그레이션 (처음부터 단계별)

- `meta.save_version`(정수). 로드 시 엔진이 현재 버전과 비교.
- **v(n)→v(n+1) 단계별 up-migration 체인**을 처음부터 구축. 여러 버전 건너뛰어도 순차 적용.
- 각 마이그레이션 = 트랜잭션 1개, 실패 시 롤백 + **전 원본 백업** 보존.
- `content_version`도 meta에 기록 — content.db 변경 영향 감지.

## 6. 무결성 (HMAC — 서명+감지)

- **서명 대상**: 핵심 상태(protagonist·npc·history·meta) 해시 → `meta.integrity_sig`.
- **키**: Rust 바이너리 내장(네이티브라 추출 난이도↑). **갱신**: 주간 커밋마다 재서명.
- **위반 시**: 로드 차단 안 함 → **'수정됨' 플래그 표시**. 온라인 요소 도입 시 서버검증으로 승격.

## 7. 다국어 (i18n) — 초반부터 (런칭 ko + en)

### 크롬 vs 콘텐츠 분리

| 구분 | 대상 | 위치/방식 |
|---|---|---|
| **UI 크롬** | 버튼·메뉴·시스템 문자열 | **Flutter 표준 i18n**(slang/ARB), Dart |
| **게임 콘텐츠** | 이벤트·대사·메시지·업적 | **content.db 번역키 테이블**, 엔진이 로케일 해결 |
| **이름(데이터)** | 선수·팀 이름 | 로케일 태그 이름풀 — 번역 아님 |
| **refs 표시명** | 팀·리그 표시명 | strings 테이블 참조 |

### 콘텐츠 번역 = 번역키 테이블 (업계 표준)

- 콘텐츠 행은 `string_id` 참조, `strings(string_id, locale, text)`가 번역 보유.
- 표준 근거: Paradox·Unity/Unreal String Tables·gettext(PO)·RimWorld 모두 키 기반.
- **AI-gen 적합**: 소스(ko) 생성 → AI가 en을 같은 테이블에 채움 → 검증기가 미번역 키 리포트.
- 엔진: `setLocale(locale)` → 이후 쿼리가 해당 언어로 해결(UI는 번역된 텍스트만 받음). meta/settings에 로케일 저장. **폴백 = ko**.

## 8. 불만 → 해소 / 폐기

| 불만 | 해소 |
|---|---|
| 파일 양 많음 | content.db 하나 |
| 수정 어려움 | DB 직접 편집, 즉시 반영 |
| 빌드 느리고 꼬임 | 빌드 소멸(import는 AI 추가 시에만) |
| DB 디버깅 어려움 | DB가 원본 truth → 대조 불필요 |
| 데이터 흩어짐 | 저장 기술 하나·접근 계층 하나 |

**폐기(프로토 대비)**: staging/deploy, build:masterdb, gen:manifest, `_manifest.json`, `_index.json`, JSON→SQLite 컴파일, entities 사전생성 선수 JSON.
