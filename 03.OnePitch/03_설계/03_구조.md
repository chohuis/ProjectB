# 구조 (계층 계약 · 엔진 API · Flutter 구조)

> 결정 기록 — 프로젝트의 가장 중요한 뼈대. 근거: [분석서](../01_분석/프로토타입_분석서.md) §3 · 요약: [00_결정_요약](00_결정_요약.md)
> 상태: **확정(초안)** · 일자: 2026-07-09
> 배경: 프로토는 이 경계가 흐릿해(로직이 스토어로 샘, NPC 상태 6곳 분산, 타입 어긋남) 개발 중 버그가 잡기 힘들었다. 정식버전은 경계를 **엄격·명시적**으로 못 박아 그 버그 클래스를 차단한다.

## 1. 계약 방향 (사용자 확정 3문장)

1. **모든 계산은 Rust 엔진에서 숨겨 처리 → 결과만 UI로.**
2. **선수 정보·이벤트 대사 등은 데이터층에서 관리.**
3. **UI는 렌더링(동작)만.**

## 2. 계층 구조

```
┌─────────────────────────────────────────────┐
│  Flutter (Dart) — UI                         │
│  · 화면 렌더 + 입력 전달만                    │
│  · 금지: 계산·판정·난수·게임 데이터 직접 보유  │
└───────────────┬─────────────────────────────┘
                │ flutter_rust_bridge (타입드, 유일 통로)
┌───────────────┴─────────────────────────────┐
│  Rust 엔진 — 유일한 권위(single authority)    │
│  · 모든 계산·판정·난수·상태 변이              │
│  · 읽기/쓰기 API로 "해결된 결과"만 반환       │
│  · 은닉 밸런스는 여기 상수(D4-2)              │
└──────┬───────────────────────┬───────────────┘
┌──────┴──────┐        ┌────────┴─────────┐
│ content.db  │        │  slot_<id>.db    │
│ 정의(콘텐츠) │        │  상태(세이브)     │
└─────────────┘        └──────────────────┘
```

## 3. 핵심 불변식 (프로토 버그 클래스 차단)

1. **UI에 로직 0** — 난수·확률·판정·집계가 UI/상태관리에 없다. → 로직이 스토어로 새던 문제 차단.
2. **UI는 데이터 직접 접근 0** — content.db도 slot.db도 UI가 직접 안 읽는다. 엔진이 **대사까지 해결된 결과**로 넘긴다. → 콘텐츠 이중 읽기 어긋남 차단.
3. **경계는 하나, 타입드** — Dart↔Rust는 frb 코드젠으로만. snake↔camel 어긋남류가 **컴파일 단계**에서 잡힘.
4. **엔진이 유일 권위** — 읽기·쓰기·계산 전부 엔진 API 경유. "스토어에서? 엔진에서?" 모호함 소멸 → 버그 위치가 항상 명확.
5. **결정적** — 엔진 난수는 worldSeed 기반 → 버그 재현 가능.

## 4. 은닉 vs 개방 밸런스 (D4-2)

| 개방 (content.db) | 은닉 (Rust 상수) |
|---|---|
| 이벤트·대사·메시지·업적 | 매치엔진 확률 테이블 |
| 이름풀·캐릭터·아크 | 성장 핵심계수(XP임계·성장곡선) |
| 선수 생성 규칙(로스터) | 평가(S~D) 판정 경계·부상 확률 |
| 표시용 라벨/설명 | 요약: **"결과를 좌우하는 확률·계수"는 전부 Rust** |

## 5. 엔진 API 표면 = 범용 쿼리 + 커맨드

| 종류 | 예 | 성격 |
|---|---|---|
| **쿼리(읽기)** | `queryPlayers(filter)`·`queryStandings(leagueId)`·`queryCareerRecord()` | 엔진이 계산·집계 끝낸 도메인 데이터 타입드 반환 |
| **커맨드(쓰기/진행)** | `advanceWeek()`·`resolveChoice(evtId, choiceId)`·`signContract(...)` | 상태 변이 + 갱신 결과 반환 |
| **매치 세션** | `startMatch()→handle`·`pitch(handle, input)` | 세션 상태를 Rust에 보유(frb 불투명 핸들), 1구 단위 |
| **진행 정지** | `advanceWeek()` → `PendingAction[]` | UI가 표시 → 유저 처리 → 커맨드로 회신 |

### "UI엔 로직 0" 실무 경계선

| 엔진이 함 | UI가 해도 됨 | UI 금지 |
|---|---|---|
| 파생값(OVR·ERA·평점·순위)·집계·판정·대사해결·난수 | 무엇을 보여줄지 선택, 배치, **엔진이 준 값으로** 정렬/필터, i18n 라벨·숫자/날짜 포맷 | 게임 값 계산, 확률·난수, 게임 판정, 화면 밖 원본 보유 |

## 6. Flutter 구조

- **상태관리 = Riverpod** — 게임 상태는 전부 Rust에. Dart는 **엔진 결과의 얇은 캐시 + UI 로컬 상태**만. async(FutureProvider/AsyncNotifier)가 frb 호출을 감쌈.
- **라우팅 = go_router** — PC/모바일 공용.
- **화면 = 기능 우선(feature-first)** + **처음부터 반응형**(데스크톱 사이드네비 / 모바일 바텀네비).
- **매치 렌더** = Flame 또는 CustomPainter (상세는 UI 기획).

### 폴더 구조 (초안)

```
onepitch/
├── app/                      # Flutter (Dart) — UI만
│   └── lib/
│       ├── main.dart
│       ├── features/         # roster·status·training·match·records·messenger …
│       │   └── <feature>/{view, provider}
│       ├── shared/           # router·공통위젯·theme·i18n(크롬)
│       └── bridge/           # frb 생성 Dart 바인딩(자동생성)
└── engine/                   # Rust crate — 게임 본체
    ├── src/
    │   ├── lib.rs            # frb export 진입점
    │   ├── api/              # 쿼리·커맨드 표면
    │   ├── sim/              # 매치·성장·npc·로스터·일정
    │   ├── data/{content.rs, slot.rs}
    │   ├── balance/          # 은닉 밸런스 상수(D4-2)
    │   └── integrity/        # HMAC
    └── content.db            # 콘텐츠 원본(번들)
```

## 7. 이 계약이 프로토 버그를 없애는 방식

| 프로토 반복 버그 | 원인 | 이 계약에서 |
|---|---|---|
| 트레이드/드래프트/FA 화면·DB 불일치 | 상태 6곳 분산·다중 쓰기 | 엔진 유일 권위, 커맨드 트랜잭션 1개 |
| 필드명 어긋남(snake↔camel) | 수기 IPC 등록 | frb 타입드 코드젠 |
| 로직이 스토어에 누적 | UI/엔진 경계 모호 | UI 로직 0 불변식 |
| 콘텐츠 표시 불일치 | 두 곳에서 콘텐츠 읽음 | 엔진이 해결된 결과만 |
