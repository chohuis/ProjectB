# 스키마 v1 동결 (구현 브리지)

> 근거: [02_데이터](02_데이터.md)(테이블 목록·원칙) · [04_게임루프](04_게임루프.md)(pending_actions) · [08_개인_재정](../02_기획/08_개인_재정.md) §5(finance) · [분석서](../01_분석/프로토타입_분석서.md) §4(ID 3중포맷 부채) · 대화 설계(2026-07-14)
> 상태: **v1 DDL 확정** — P7 스파이크(2026-07-14, [08_P7_체크리스트](08_P7_체크리스트.md))에서 rusqlite CRUD·FK·PRAGMA 실측 통과로 동결
> 목적: [02_데이터](02_데이터.md)의 테이블 목록을 **구현 착수 가능한 구체 DDL**로 굳힌다. `meta.save_version = 1`의 기준선이자 마이그레이션 체인의 출발점.

## 1. 원칙 ([02_데이터](02_데이터.md) 계승)

- **평칼럼 vs JSON**: 조회·FK·인덱스 대상은 평칼럼, 중첩·가변 구조는 JSON(TEXT) 칼럼.
- **능력치 동거**: `stats`·`xp`·`live_state`·`finance`는 JSON 1칼럼 — 스탯/필드 추가 시 스키마 패치 불필요([02_데이터](02_데이터.md) §4).
- **content.db** = read-only 번들(정의), **slot.db** = 파일=슬롯(상태).

## 2. ID 규약 — 프로토 3중포맷 부채 청산

프로토는 `TEAM_*`/`PKT_*`/`CLUB_*` 3중 포맷으로 고생([분석서](../01_분석/프로토타입_분석서.md) §4). 정식은 **단일 포맷**으로 고정.

- **형식**: `<type>:<slug>` (예: `league:kbl`, `team:seoul_royals`, `school:hanseong_hs`, `pitch:four_seam`, `evt:e0001`, `char:c0001`).
- **NPC**: worldSeed 결정적 생성이므로 `npc:<worldSeed>_<seq>` — 저장 후엔 slot.db 내 불변 PK.
- **단일 출처**: content.db `refs`(leagues/teams/schools) = ID 정본. 부팅 시 무결성 assert([03_구조](03_구조.md) §3).

## 3. content.db DDL (v1, 발췌)

```sql
-- 콘텐츠 정의 (read-only 번들)
CREATE TABLE events (
  id TEXT PRIMARY KEY, stage TEXT, week INTEGER, type TEXT,
  urgency TEXT,                      -- 긴급/비긴급 (04_게임루프 §4)
  trigger TEXT, body TEXT            -- JSON: 조건·title/lines/choices
);
CREATE TABLE pitch_types (
  id TEXT PRIMARY KEY, family TEXT,  -- 패스트볼/브레이킹볼/오프스피드/특수
  name TEXT, meta TEXT               -- JSON
);
CREATE TABLE characters (id TEXT PRIMARY KEY, name TEXT, role TEXT, arcs TEXT);
CREATE TABLE personality_rules (context TEXT PRIMARY KEY, trait_weights TEXT);
CREATE TABLE narrative_templates (id TEXT PRIMARY KEY, category TEXT, template TEXT, variants TEXT);
CREATE TABLE achievements (id TEXT PRIMARY KEY, category TEXT, condition TEXT, meta TEXT);
CREATE TABLE name_pools (id TEXT PRIMARY KEY, locale TEXT, kind TEXT, names TEXT);
CREATE TABLE generation_rules (league_id TEXT PRIMARY KEY REFERENCES leagues(id), rules TEXT);
CREATE TABLE world_config (key TEXT PRIMARY KEY, value TEXT);   -- canonical_seed 등 생성 설정 ([07_데이터관리](07_데이터관리.md))

-- refs (ID 정본)
CREATE TABLE leagues (id TEXT PRIMARY KEY, meta TEXT);
CREATE TABLE teams   (id TEXT PRIMARY KEY, league_id TEXT REFERENCES leagues(id),
                      color TEXT, meta TEXT);                     -- 08_팀_컬러
CREATE TABLE schools (id TEXT PRIMARY KEY, region TEXT, meta TEXT);
CREATE TABLE team_traits  (team_id TEXT PRIMARY KEY REFERENCES teams(id),
                           philosophy TEXT, resource TEXT, status TEXT);  -- 05_팀특성
CREATE TABLE team_history (team_id TEXT PRIMARY KEY REFERENCES teams(id),
                           founded_year INTEGER, budget INTEGER,
                           rivals TEXT, season_ranks TEXT, titles TEXT);  -- 06_히스토리 (JSON 배열)

-- 밸런스·번역
CREATE TABLE balance_open (key TEXT PRIMARY KEY, value TEXT);   -- 개방 밸런스만 (D4-2)
CREATE TABLE strings (string_id TEXT, locale TEXT, text TEXT, PRIMARY KEY(string_id, locale));
```

- **은닉 밸런스는 여기 없음** — [05_밸런스](05_밸런스.md)의 상수는 전부 Rust 상수(D4-2).

## 4. slot.db DDL (v1, 발췌) — 5그룹

```sql
-- 메타
CREATE TABLE meta (
  save_version INTEGER, content_version INTEGER, world_seed INTEGER,  -- world_seed=세이브별(이후 유입 생성용). 초기 로스터는 content.db canonical_seed ([07_데이터관리](07_데이터관리.md) §2)
  stage TEXT, playtime INTEGER, current_day INTEGER, integrity_sig TEXT
);

-- 주인공 (1행)
CREATE TABLE protagonist (
  id TEXT PRIMARY KEY, name TEXT, handedness TEXT, archetype TEXT,  -- 07_주인공생성
  stats TEXT, xp TEXT, live_state TEXT, finance TEXT,               -- JSON 동거
  pitches TEXT, contract TEXT, injury TEXT                          -- JSON
);

-- 선수 (NPC 전부의 단일 정본)
CREATE TABLE npc (
  id TEXT PRIMARY KEY, name TEXT, team_id TEXT, position TEXT, age INTEGER,
  is_named INTEGER, retired INTEGER, form REAL, personality TEXT,
  stats TEXT, xp TEXT, live_state TEXT, pitches TEXT                -- JSON 동거 (투수만 pitches)
);

-- 관계
CREATE TABLE relationships (
  npc_id TEXT PRIMARY KEY REFERENCES npc(id),
  value INTEGER, arc_stage INTEGER                                 -- -100~100, 0~3 (01_캐릭터 §5·5-1)
);

-- 시즌 (휘발 — 시즌 종료 시 압축→리셋)
CREATE TABLE season_meta   (key TEXT PRIMARY KEY, value TEXT);
CREATE TABLE schedule      (game_id TEXT PRIMARY KEY, day INTEGER, home TEXT, away TEXT, result TEXT);
CREATE TABLE standings     (team_id TEXT PRIMARY KEY, w INTEGER, l INTEGER, t INTEGER, rank INTEGER);
CREATE TABLE season_stats  (player_id TEXT, week INTEGER, line TEXT, PRIMARY KEY(player_id, week));
CREATE TABLE pending_actions (                                     -- 04_게임루프 §5
  id TEXT PRIMARY KEY, type TEXT, urgency TEXT, created_day INTEGER, payload TEXT
);
CREATE TABLE inbox (id TEXT PRIMARY KEY, kind TEXT, urgency TEXT, read INTEGER, day INTEGER, body TEXT);

-- 기록 (영구 · append-only)
CREATE TABLE career_history (season INTEGER PRIMARY KEY, line TEXT);   -- 주인공 시즌 라인
CREATE TABLE game_log       (game_id TEXT PRIMARY KEY, season INTEGER, detail TEXT);  -- 압축 안 함
CREATE TABLE league_transactions (id TEXT PRIMARY KEY, day INTEGER, kind TEXT, detail TEXT);
CREATE TABLE history_standings   (season INTEGER, team_id TEXT, rank INTEGER, PRIMARY KEY(season, team_id));
CREATE TABLE history_leaders     (season INTEGER, category TEXT, player_id TEXT, PRIMARY KEY(season, category));
CREATE TABLE achievement_progress (ach_id TEXT PRIMARY KEY, counter INTEGER, achieved_day INTEGER);  -- 04_업적
```

- **is_named/retired/read/is_...** = INTEGER 0/1 (SQLite bool 관례).
- **JSON 칼럼 내부 스키마**(stats 9키, finance 필드 등)는 구현 단계 상세 — 스키마 패치 없이 진화([02_데이터](02_데이터.md) §4).

## 5. 버전 · 무결성

- `meta.save_version = 1` 기준선. **v1→v2 up-migration 체인을 처음부터** 구축([02_데이터](02_데이터.md) §5) — 지금은 v1뿐이지만 마이그레이션 프레임을 먼저 만든다.
- `meta.integrity_sig` = 핵심 테이블(protagonist·npc·history·meta) 해시 HMAC([02_데이터](02_데이터.md) §6). 주 진행마다 재서명.
