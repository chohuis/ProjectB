# P7 검증 스파이크 체크리스트

> 근거: [01_환경](01_환경.md) §3-2·§7 · [06_스키마](06_스키마.md) §6 · [00_결정_요약](00_결정_요약.md) G · 대화 설계(2026-07-14)
> 상태: **Windows PC 트랙 실행 완료(2026-07-14)** — #1~6·9(빌드)·10·11 통과. Android 실기기(#7~8)·Steam 실클라이언트 연동(#9 런타임)·Mac/Linux(#4 일부)는 실기기·별도 머신 필요로 보류. 결과는 §4.
> 목적: Flutter+Rust(frb)+Android+Steam은 이 프로젝트가 처음 다루는 조합. 문서·기획으로는 더 못 좁히는 "실제로 되는가"만 남은 항목을 실측으로 확정한다.

## 1. 스코프 · 원칙

- **시간박스 1~3일**. 스파이크 산출물(테스트 프로젝트)은 검증 후 **버림** — 실제 구현의 시작점이 아니라 리스크 제거용 1회성 코드.
- **목적 = "되는가" 확인이지 "잘 만드는가"가 아니다.** 최소 hello-world 수준으로만 각 항목을 통과시킨다.
- 각 항목은 **성공기준**과 **실패 시 폴백**을 미리 정해둔다 — 실측 도중 판단하지 않는다(사후 합리화 방지).
- 순서는 **의존성 순**(앞 항목이 안 되면 뒤 항목은 의미 없음) — §2.

## 2. 실측 순서

```
1. 툴체인 설치(Flutter SDK·Rust·cargo-ndk·Android SDK/NDK·frb codegen 도구)
2. frb 최소 hello-world (Dart→Rust 함수 호출, 코드젠 확인) — PC
3. rusqlite 최소 CRUD + PRAGMA(foreign_keys, journal_mode) — PC
4. PC 데스크톱 빌드 3종(Windows/Mac/Linux) — cargo build + flutter build 성공
5. content.db 번들링 + 오픈 — PC(리소스 직접 오픈)
6. Android 크로스컴파일 — cargo-ndk로 .so(arm64-v8a 최소 1종)
7. content.db 번들 → Android 내부저장소 추출 + 오픈 — path_provider
8. Android 실기기/에뮬레이터에서 frb + rusqlite 실동작
9. flutter_steamworks 연동 — PC만(Steam 클라이언트 감지 + 더미 API 호출 1개)
10. 논블로킹 확인 — 무거운 Rust 연산을 frb async로 호출 시 UI 프리즈 없는지
11. HMAC/암호화 크레이트 크로스컴파일 확인 — Android .so에도 포함되는지
```

## 3. 항목별 성공기준 · 실패 시 폴백 · 실측 결과

| # | 항목 | 성공기준 | 실패 시 폴백 | 결과(2026-07-14) |
|---|---|---|---|---|
| 1 | 툴체인 설치 | Flutter/Rust/cargo-ndk/Android NDK 버전 조합이 문서화된 절차로 재현 가능 | 버전 조합 재탐색, 실패해도 스택 자체를 바꾸지 않음(설치 이슈는 버전 핀으로 해결) | ✅ 통과 — §4 버전표 |
| 2 | frb hello-world | Dart에서 Rust 함수 호출·반환값 수신, 코드젠 명령 1회로 성공 | frb 버전 다운/업 조정. 근본 실패 시에만 [01_환경](01_환경.md) §3-2 재검토 트리거 | ✅ 통과 — Dart `greet()`→Rust 호출·반환값 일치, 헤드리스 `flutter test`로 검증 |
| 3 | rusqlite CRUD+PRAGMA | 테이블 생성·삽입·조회·FK 제약 위반 감지 확인 | 타입 매핑만 [06_스키마](06_스키마.md) 조정, 아키텍처 변경 아님 | ✅ 통과 — FK 위반 정상 차단, `PRAGMA foreign_keys` 확인, `bundled` 기능(소스내장 SQLite)으로 빌드 |
| 4 | PC 3종 빌드 | 3플랫폼 모두 실행 파일 생성+정상 기동 | 특정 OS만 실패 시 해당 OS 런처 옵션(예: Linux는 별도 배포판 이슈) 개별 조사 | **Windows만 ✅**(`p7_spike.exe` 빌드 성공). Mac/Linux는 이 실행 환경이 Windows뿐이라 미검증 — 별도 머신/CI 필요, 열린 세부로 이월 |
| 5 | content.db PC 오픈 | 번들 리소스에서 읽기 성공 | 경로 규약만 조정 | ✅ 통과 — Flutter 에셋 번들(`data/flutter_assets/assets/content.db`)을 Rust가 read-only로 열고 쿼리 성공 |
| 6 | Android cargo-ndk | `.so` 생성 + APK에 정상 포함 | NDK 버전 핀 재조정. 반복 실패 시 ABI 범위 축소(우선 arm64-v8a만) | ✅ 통과 — `flutter build apk`가 cargokit 경유로 arm64-v8a·x86_64용 `.so` 자동 빌드, APK 내 포함 확인(`unzip -l`) |
| 7 | content.db Android 추출 | 첫 실행 시 내부저장소로 추출 후 오픈 성공 | 추출 트리거 조건(`content_version` 비교) 로직 재점검 | ⏸ **보류** — 에뮬레이터/실기기 없어 미검증. 사용자 실제 Android 기기에서 확인 예정 |

| 10 | 논블로킹 | 수백 ms급 Rust 연산 중 Flutter UI가 계속 반응 | frb async 호출 방식(별도 isolate/thread) 조정 | ✅ 통과 — Rust 500ms sleep을 async로 호출하는 동안 Dart 50ms 주기 타이머가 계속 발화함을 확인(이벤트루프 안 막힘) |
| 11 | HMAC 크로스컴파일 | 사용 크레이트가 Android NDK 타깃으로 정상 빌드 | 대체 크레이트 탐색(암호화 알고리즘 자체는 변경 없음) | ✅ 통과(Windows) — `hmac`+`sha2` 크레이트 정상 빌드·결정적 동작 확인. Android는 별도 미검증이나 순수 Rust 크레이트라 §6이 이미 증명한 크로스컴파일 경로 리스크 낮음 |

## 4. 실측 결과 (2026-07-14, Windows PC 트랙)

### 4-1. 버전 고정표

| 구성요소 | 버전 |
|---|---|
| Flutter | 3.38.4 (stable) |
| Rust / Cargo | 1.95.0 |
| flutter_rust_bridge / codegen | 2.12.0 |
| cargo-ndk | 4.1.2 |
| JDK | Microsoft Build of OpenJDK 17.0.19.10 |
| Android SDK Platform | 34(수동설치) + 36·33(Gradle 자동보강) |
| Android Build-Tools | 34.0.0(수동) + 35.0.0(Gradle 자동보강) |
| Android NDK | 27.3.13750724(수동) + 28.2.13676358(Gradle이 선호해 자동설치) |
| flutter_steamworks | 0.0.4 |
| rusqlite | 0.31 (`bundled` feature) |

- Gradle이 프로젝트에 버전을 안 맞춰도 필요 컴포넌트(NDK/Build-Tools/Platform/CMake)를 **첫 빌드 시 자동 보강**한다 — 사전에 정확한 버전을 다 맞출 필요는 없다는 뜻, 다만 그만큼 첫 Android 빌드는 네트워크·시간이 더 든다(이번 첫 APK 빌드 ~15분).

### 4-2. 발견한 실전 이슈 2건 (둘 다 해결됨)

1. **Windows MAX_PATH(260자) 제한** — 프로젝트를 깊은 경로(세션별 임시폴더 등)에 두면 CMake/MSBuild가 중간 산출물 경로를 못 만들어 빌드 실패. **해결**: 프로젝트를 짧은 경로에 둔다(`C:\` 바로 아래 등). 실제 리포지토리 구조([03_구조](03_구조.md) `app/`+`engine/`)는 이미 짧은 경로라 실서비스엔 영향 없음 — 스파이크 자체의 임시 경로 때문에만 발생했던 문제.
2. **`flutter_steamworks` 0.0.4가 한국어(비영어) 로케일 Windows에서 빌드 실패** — 패키지의 C++ 소스(`flutter_steamworks_plugin.cpp`)에 BOM 없는 UTF-8 문자열이 들어있는데 MSVC가 시스템 코드페이지(949)로 잘못 해석해 컴파일 에러. **해결**: 프로젝트 `windows/CMakeLists.txt`의 `APPLY_STANDARD_SETTINGS`에 `/utf-8` 컴파일 플래그 추가로 우회 성공. **실구현 시 반드시 이 플래그를 프로젝트 표준 CMake 설정에 포함할 것** — 패키지 자체 수정이 아니라 우리 프로젝트 설정으로 해결 가능하므로 [01_환경](01_환경.md) §3-2 Steam 결론(flutter_steamworks 채택)은 유지.
3. **`flutter test`는 빌드된 앱과 별개 프로세스** — 네이티브 `.dll`을 자동으로 못 찾는다(빌드된 `Release/` 폴더에서 프로젝트 루트로 수동 복사 필요). 실제 `flutter run`/설치된 앱 실행 시에는 빌드 시스템이 자동 배치하므로 최종 사용자 경험엔 영향 없음 — **CI에서 Dart 단위테스트를 돌릴 때만** 챙겨야 할 스텝(예: 빌드 후 DLL 복사를 CI 스크립트에 포함).

### 4-3. 검증 산출물

- 스파이크 프로젝트(`p7_spike`, Windows 로컬 임시 경로) — 원칙대로 **버림**, 리포지토리에 남기지 않음.
- **[06_스키마](06_스키마.md) v1 최종 확정** — rusqlite CRUD·FK·PRAGMA 실측 통과로 DDL 후보 → 동결.
- **막힌 항목 있으면 해당 절만 재오픈** — 전면 재설계 아님(§3 폴백란이 원칙). 이번엔 막힌 항목 없음(#7~8·Steam 런타임은 "미검증"이지 "실패"가 아님).

## 5. 스파이크 범위 밖 (착각 방지)

- 실제 게임 로직·화면 구현·콘텐츠 저작·수치 밸런싱은 여기서 안 함(전부 이후 구현 단계).
- UI 디테일(레이아웃·애니메이션)도 범위 밖 — hello-world 수준 화면 1개면 충분.
- 172팀 시드 파일 실제 생성([07_데이터관리](07_데이터관리.md) §3-3)도 범위 밖 — 스키마만 확정되면 됨.
