## 2. 조립 방식 — 변수 슬롯 템플릿 + 조건별 문장 뱅크

[03_설계/02_데이터](../../03_설계/02_데이터.md)의 기존 원칙("AI 대량생성은 저작 단계에서만, 런타임엔 Rust가 메모리 로드")을 그대로 따른다.

| 방식 | 채택 여부 |
|---|---|
| **변수 슬롯 템플릿**("{투수}가 {이닝}이닝 {실점}실점으로 막아냈다") | ✅ 기본 골격 |
| **조건별 문장 뱅크**(같은 상황도 뼈대 여러 개 중 랜덤 선택) | ✅ 바리에이션 확보용, 위와 결합 |
| **런타임 AI 생성** | ❌ 배제 — 오프라인 플레이 불가·비결정적 결과·엔진 유일권위 원칙과 충돌 |

**"저작은 AI로 풍부하게, 실행은 결정적으로 가볍게"** — 저작 단계(오프라인)에서 AI로 템플릿·문장뱅크를 대량 생성해 content.db에 import, 런타임은 순수 조회+슬롯채우기만 한다.

## 3. 다국어 연동

기존 i18n 설계(strings 테이블: `string_id, locale, text`)에 그대로 얹는다.

| 요소 | 처리 |
|---|---|
| 템플릿 문장 뼈대 | strings 테이블에 `string_id`로 저장, 언어별(ko/en) 버전 존재 |
| 변수 슬롯 값(선수 이름 등) | 이름풀=로케일 데이터(번역 아님, 기존 데이터 설계 결정 재사용) |
| 숫자·스탯 값 | 그대로 결합(번역 불필요) |
| 한국어 조사(은/는/이/가 등 받침 여부) | 별도 규칙 필요 — §5 열린 세부로 이월 |

## 4. 반복 방지

| 장치 | 방식 |
|---|---|
| **문장뱅크 규모** | 저작 단계에서 상황당 충분히 많은 후보(예: 10개 이상) AI 생성 |
| **최근 사용 회피** | [02_이벤트](02_이벤트.md) 쿨다운 개념 재사용 — 최근 사용한 템플릿은 이번 선택 풀에서 잠시 제외, 완전 소진되면 다시 전체 허용 |
