# 콘텐츠 — 개요 · 로드맵

> 근거: [게임 개요](../00_게임_개요.md) · 지금까지 만든 모든 트리거·훅(07_매치_엔진 라이벌매치업·개인기록근접, 05_히스토리_엔딩 관계로그·수상로그, 팀_특성_시스템 특성전용이벤트, 08_부상_시스템 전조경고, 06_시장_계약 방출경고·FA오퍼, 04_프로_커리어 갈림길 등)
> 상태: **로드맵 6/6 전부 구조 확정(초안) — 콘텐츠 시스템 완주**
> 목적: 지금까지 시스템 설계에서 열어둔 모든 트리거·훅을 실제 콘텐츠 형태로 채운다. 프로토(`CONTACT_*`·`SCRIPT_*`·`EVT_*`·`MSG_*`/`DEC_*`·`ACH_*`)는 카테고리 존재 참고만, 구조는 백지 재설계.

## 의존 순서 (왜 이 순서인가)

```
1. 캐릭터        ← 모든 콘텐츠가 참조하는 기반 데이터(누가 있는가)
2. 이벤트        ← 트리거·조건·선택지의 핵심 구조(콘텐츠 시스템의 척추)
3. 메시지·알림   ← 이벤트 트리거 패턴을 단순 정보성 채널로 재사용
4. 업적          ← 트리거·조건 재사용, 독립적으로 설계 가능
5. 뉴스·미디어   ← 세계관 풍미, 기계적 의존도 낮음
6. 서술 템플릿   ← 위 전부를 procedural text로 엮는 마무리 규칙
```

- 1(캐릭터)이 먼저인 이유: 이벤트·뉴스 전부 "누가 말하는가"가 없으면 설계할 수 없음.
- 2(이벤트)가 척추인 이유: 트리거(언제)·조건(누구에게)·선택지(무엇을)의 3요소가 이후 모든 카테고리에 재사용됨.
- **메신저(카톡형 지속 대화 채널)는 제거** — 관계 서사는 캐릭터 성격축·관계수치를 바탕으로 이벤트가 담당.
- **컷신·연출은 제거** — 은퇴식 연출은 이미 [05_히스토리_엔딩](../05_히스토리_엔딩.md) §4가 소유(서술 템플릿이 텍스트 조립), 수상 관련은 [04_업적](04_업적.md)+05_히스토리_엔딩 수상기록로그로 이미 커버돼 별도 카테고리로 남길 내용이 없었음.
- 6(서술 템플릿)이 마지막인 이유: 07_매치_엔진 결과 요약, 05_히스토리_엔딩 커리어 내레이션·은퇴식 연출처럼 여러 시스템의 출력을 텍스트로 조립하는 규칙이라 나머지가 먼저 있어야 함.

## 설계 원칙 (게임 개요에서 상속)

- **1인칭 원칙** — 모든 대사·메시지·이벤트는 "나에게 일어나는 일"로 서술(2인칭 관찰자 시점 아님).
- **가상 세계** — 실제 인물·팀·사건 인용 금지. 문서 본문에도 예시는 전부 가상 서술로만 작성.
- **content.db 소속** — [03_설계/02_데이터](../../03_설계/02_데이터.md) 3계층 원칙에 따라 콘텐츠 전부 content.db(직접저작+AI import), 상태/진행은 slot.db.
- **느슨한 기대치 압박** — 이벤트·메시지가 강제 실패를 유도하지 않음. 선택의 무게는 서사적이지 시스템적 게임오버가 아님.
