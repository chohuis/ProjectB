## 1. 정보 원천·범위 — 상세도는 데이터 부족이 아니라 UX 선택

[리그팀/00_개요](../리그팀/00_개요.md) §4-1에 따르면 **현재 전 리그가 풀 시뮬로 확정**돼 있어(하이브리드는 해외리그 DLC 때만 재적용 검토), 데이터 자체는 어디든 다 존재한다. 따라서 아래 상세도 차등은 **데이터 부족 때문이 아니라 정보 과부하 방지 + 주인공 중심성 유지를 위한 의도적 UX 선택**이다.

| 범위 | 데이터 출처 | 뉴스 상세도 |
|---|---|---|
| **소속팀 소식** | 주인공 소속팀 경기 데이터(풀 시뮬로 이미 존재) | 상세(경기결과+코멘트) |
| **라이벌·동료 소식** | [01_캐릭터](01_캐릭터.md) 상태·마일스톤 | 상세 |
| **시장 소식**(타 팀 트레이드·FA) | [06_시장_계약](../06_시장_계약.md) NPC간 캐던스(연2회) | 중간(거래 사실만, 배경설명 생략) |
| **인접 권역 소식** | 풀 시뮬 데이터 존재하나 **의도적으로 요약만 노출** | 요약 수준 |
| **먼 리그 소식** | 풀 시뮬 데이터 존재하나 **의도적으로 헤드라인만 노출** | 헤드라인만("OO리그 1위 등극") |

- 해외 리그(DLC)로 하이브리드 원칙이 재적용되면, 그때는 "결과만" 구간이 실제 데이터 제약과 일치하게 된다 — 지금은 UX 선택이지만 구조는 그대로 재사용 가능.

## 2. 작성 주체·톤 — 중립 기사체로 통일

[03_메시지_알림](03_메시지_알림.md)과 같은 이유로 개별 기자의 문체를 저작하지 않는다.

| 항목 | 정리 |
|---|---|
| 기사 텍스트 | 중립 기사체(신문투 서술)로 통일 — [06_서술_템플릿](06_서술_템플릿.md)이 조립 담당 |
| 기자 NPC의 역할 | 텍스트 작성자로서의 개성은 없음. [01_캐릭터](01_캐릭터.md) 성격축은 인터뷰 요청 이벤트([02_이벤트](02_이벤트.md) 확률형 플레이버)처럼 "얼마나 공격적으로 질문하는지" 정도에만 쓰임 |

## 3. 트리거 방식 — 혼합

| 뉴스 유형 | 메커니즘 |
|---|---|
| 소속팀·라이벌·동료 소식 | **콜백** — 해당 사건 발생 즉시(경기결과·마일스톤) |
| 시장 소식(트레이드·FA) | **콜백** — [06_시장_계약](../06_시장_계약.md) NPC간 캐던스 확정 시 |
| 인접권역·먼 리그 소식 | **폴링(주간 다이제스트)** — 배경 시뮬 결과를 매주 한 번 요약해 발행 |

[02_이벤트](02_이벤트.md) §1의 폴링/콜백 메커니즘을 그대로 재사용, 새 트리거 아키텍처 없음.
